# Contributing to Codex Nexus

First of all, thank you for your interest in contributing to **Codex Nexus**.

Our goal is to build a modular, privacy-first platform that helps people organize, connect, and discover knowledge while maintaining ownership of their data. Every contribution, whether it is code, documentation, bug reports, or ideas, is appreciated.

---

# Code of Conduct

Please be respectful, constructive, and collaborative.

We welcome contributors from all backgrounds and encourage open discussions based on mutual respect and technical merit.

---

# Development Principles

When contributing to Codex Nexus, please follow these principles:

* Keep the architecture modular.
* Prefer readability over cleverness.
* Write clear and maintainable code.
* Document important decisions.
* Respect the project's privacy-first philosophy.
* Avoid introducing unnecessary dependencies.

---

# Getting Started

1. Fork the repository.
2. Create a feature branch.

```text
git checkout -b feature/your-feature-name
```

3. Make your changes.
4. Test your changes.
5. Commit using a meaningful commit message.
6. Push your branch.
7. Open a Pull Request.

---

# Commit Messages

Please use descriptive commit messages.

Examples:

```text
Add document indexing engine

Improve search performance

Fix metadata parser

Update architecture documentation
```

Avoid generic messages such as:

```text
Update

Fix

Changes
```

---

# Pull Requests

A good Pull Request should:

* Focus on a single feature or fix.
* Include a clear description.
* Update documentation when necessary.
* Keep changes as small and focused as possible.

---

# Documentation

Documentation is considered part of the project.

When introducing significant architectural or design changes:

* Update the relevant documentation.
* Create or update an Architecture Decision Record (ADR) when appropriate.

---

# Reporting Issues

When reporting a bug, please include:

* Description
* Steps to reproduce
* Expected behavior
* Actual behavior
* Operating system
* Python version (if applicable)

Feature requests should explain:

* The problem being solved.
* The proposed solution.
* Possible alternatives.

---

# Coding Style

Current development follows these guidelines:

* Python best practices (PEP 8)
* Clear naming conventions
* Small, reusable functions
* Modular architecture
* Type hints where appropriate

Code should be easy to understand and maintain.

---

# Philosophy

Codex Nexus is more than software.

It is a platform built on the belief that knowledge should remain under the user's control.

Every design decision should support:

* Privacy
* Transparency
* Modularity
* Extensibility
* Long-term maintainability

---

Thank you for helping build Codex Nexus.

**Where knowledge comes together.**
